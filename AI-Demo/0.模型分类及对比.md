# 模型分类及对比 (2024-2025)

本文档汇总了截止 2025 年当前时间节点的主流 AI 模型，涵盖大语言模型 (LLM)、视觉生成模型、多模态理解模型以及 OCR/文档理解模型。

# 模型查找社区

https://www.modelscope.cn/models

尽量使用比较成熟,使用频率较高的模型, 否则需要自己进行测试.

## 1. 大语言模型 (Large Language Models, LLMs)

LLM 是当前 AI 领域的核心，分为闭源商业模型和开源模型两大阵营。

### 1.1 闭源商业模型 (State-of-the-Art)

| 模型名称 | 开发商 | 估计参数量 | 上下文窗口 (Context Window) | 核心特点 | 适用场景 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **GPT-4o** | OpenAI | 未公开 (est. 1.8T MoE) | 128k | 综合能力最强，原生多模态 (Omni)，响应速度极快，语音/视觉/文本实时交互 | 复杂推理、多模态交互、代码生成、通用助手 |
| **Claude 3.5 Sonnet** | Anthropic | 未公开 | 200k | 逻辑推理、代码能力超越 GPT-4o，在 Artifacts 交互界面表现极佳，更具"人性化" | 编程辅助、长文档分析、创意写作 |
| **Gemini 3 Pro** | Google | 未公开 | 1M - 2M | 超长上下文 (Infinite Context)，原生多模态，能够处理海量文档或长视频 | 长视频理解、百万字文档分析、代码库检索 |
| **o1-preview (Strawberry)** | OpenAI | 未公开 | 128k | 强化学习训练 (CoT)，具备自我反思能力，推理能力大幅提升 (数理化) | 复杂数学证明、科学研究、高难度逻辑推理 |

### 1.2 开源/开放权重模型 (Open Weights)

| 模型名称 | 开发商 | 参数量 (Parameters) | 上下文窗口 | 核心特点 | 备注 |

## 3. 多模态与 OCR 模型 (Multimodal & OCR)

### 3.1 视觉-语言理解模型 (VLM / LMM)

这类模型能"看懂"图片并进行对话。

| 模型名称 | 开发商 | 架构/特点 | 优势领域 |
| :--- | :--- | :--- | :--- |
| **GPT-4o / Gemini 1.5** | OpenAI / Google | 原生多模态 | 通用视觉理解最强，OCR 准确率极高，能理解图表、幽默、空间关系 |
| **Qwen2-VL** | Alibaba Cloud |基于 Qwen2 | 开源最强 VLM 之一，支持任意分辨率输入，视频理解能力强 |
| **InternVL 2** | OpenGVLab | ViT + LLM | 性能对标闭源模型，在文档图表理解上表现优异 |
| **Yi-Vision** | 01.AI | - | 中文场景下的图像理解与描述能力出色 |

### 3.2 专用 OCR 与文档解析模型

专注于将图像中的文字提取并结构化，包含传统 OCR 升级版和新兴的多模态文档大模型。

| 模型名称 | 开发商/机构 | 参数量/类型 | 核心功能 | 备注 |
| :--- | :--- | :--- | :--- | :--- |
| **DeepSeek-OCR** | DeepSeek | ~3B | 端到端文档解析，支持 Markdown 输出，复杂公式/表格还原度极高 | 采用视觉压缩技术，性价比极高，适合论文/财报 |
| **GOT-OCR 2.0** | StepFun (阶跃星辰) | 580M | 统一 OCR 模型，支持纯文本、数学公式、乐谱、几何图形识别 | 开源 (Apache-2.0)，参数小，适合本地部署 |
| **TextMonkey** | HUST / Kingsoft | ~10B (基于 Qwen/Yi) | 专注于"看字"的多模态大模型，支持超高分辨率输入，擅长文档问答 | Monkey 系列升级版，解决了小文字识别问题 |
| **PaddleOCR v4** | Baidu | 轻量级系列 | 工业级 OCR 系统，检测+识别两阶段，支持 80+ 语言 | 工业界落地首选，速度快，生态极其成熟 |
| **Surya** | Vik Paruchuri | Transformer | 高精度文档 OCR 与版面分析，支持 90+ 语言，行级文本检测 | 替代 Tesseract 的现代化选择，适合扫描件 |
| **腾讯混元 (Hunyuan)** | Tencent | 闭源 API | 腾讯云 OCR 能力的集大成者，结合大模型语义理解，手写体/模糊字识别强 | 微信/QQ 扫一扫背后的技术，中文识别鲁棒性强 |
| **小红书 (Red OCR)** | Xiaohongshu | 内部/技术分享 | 针对移动端、复杂背景、艺术字体的识别优化，图文理解能力强 | 在多模态榜单 (如 OmniDocBench) 表现优异 |
| **Nougat** | Meta | Transformer (Swin) | 专门针对科学文献的 PDF 转 Markdown，公式识别准确 | 适合学术论文转换，但通用性不如 GOT |

### 3.3 文档解析工具与框架 (Document Parsing Tools)

这些通常不是单一的端到端模型，而是结合了 OCR、版面分析 (Layout Analysis) 和 LLM 修正的综合工具/Pipeline。

| 工具名称 | 核心组件/技术路线 | 功能特点 | 适用场景 |
| :--- | :--- | :--- | :--- |
| **MinerU** | Magic-PDF + PDF-Extract-Kit | 上海 AI Lab 开源，专注于将 PDF 转换为高质量 Markdown，针对学术文档优化，表格/公式还原度高 | RAG 数据清洗，学术论文预处理 |
| **Doc2X** | 商业闭源 (多模型集成) | 集成多种 OCR 引擎与 LLM，提供高精度的 PDF 转 Word/Markdown/LaTeX，擅长复杂排版 | 科研人员，需要高精度文档转换的用户 |
| **OmniParse** | 视觉模型 + 结构化解析 | 旨在将所有非结构化数据 (PDF/图/音视频/网页) 转化为结构化数据，专为 RAG 优化 | 构建多模态知识库 |
| **LLM-Aided OCR** | OCR + LLM Correction | 这是一个技术路线而非单一产品，指利用 LLM (如 GPT-4) 对 OCR 产生的乱码/错误进行后处理修正 | 提升传统 OCR 引擎的最终输出质量 |
| **docTR** | PyTorch/TF + ResNet/VGG | Mindee 开源的端到端 OCR 库，包含文本检测 (DBNet) 和识别 (CRNN/SAR) 模块，支持微调 | 开发者构建自定义 OCR 流程 |
